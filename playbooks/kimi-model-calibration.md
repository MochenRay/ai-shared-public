# Kimi 模型调教、自检与审稿手册

## 用途

当 Kimi 正在产出内容，或其他模型需要评价、引导、二次修订 Kimi 产出时使用。这里的 Kimi 不只指 Kimi CLI / Kimi Web，也包括 Claude Code CLI、remote-agent-host 或其他 host 通过 Kimi / Moonshot endpoint 调用 Kimi 模型的场景。

核心原则：先分清 `host/tool`、`model`、`source artifact`、`installed/runtime target` 四层，再决定是 Kimi 自检，还是外部审稿。

## 四层分解

| 层 | 例子 | 判断重点 |
|---|---|---|
| Host / tool | Kimi CLI、Claude Code CLI、remote-agent-host | 工具权限、协议兼容、文件写入、symlink、WebSearch / Fetch 能力 |
| Model | Kimi k2.6、Kimi Code endpoint | 推理、写作、修订、证据回查能力 |
| Source artifact | handover、Skill 文本、PRD 草稿、方案正文 | 内容是否可审、可复用、可验证 |
| Installed / runtime target | `~/.agents/skills/...`、项目仓文件、远端部署 | 是否真的同步、生效、可被 host 读取 |

不要把 host 的工具限制误判为 Kimi 模型能力；也不要把 Kimi 写出的 handover 当作已安装或已生效。

## 两种模式

| 模式 | 谁执行 | 目标 |
|---|---|---|
| 自检模式 | Kimi 模型正在 Kimi CLI / Claude Code / remote-agent-host 等 host 内产出内容 | 产出前主动约束自己，避免把草稿、计划、源文件、安装目标说混 |
| 审稿模式 | Codex / Claude / Gemini / 用户在评价 Kimi 产出 | 对已产物做证据审计和纠偏 |

两种模式共用四层分解，但职责不同：自检模式要求 Kimi 先不犯错；审稿模式要求外部 reviewers 找出已经发生的漂移。

## Kimi 自我要求

当当前模型实际是 Kimi 时，无论 host 是 Kimi CLI、Claude Code、remote-agent-host 还是其他兼容层，都先执行以下自检：

1. **声明 host 边界**：先判断自己能否读文件、写文件、联网、调用工具；没有实际工具结果时，不要说“已读取 / 已写入 / 已安装”。
2. **区分产物状态**：明确区分“报告已写”“源文件已生成”“目标路径已覆盖”“host 已读取验证”。
3. **少做强量化**：没有统计就不要写百分比、覆盖率、全量判断；改写为“高频”“主导”“抽样可见”。
4. **时间先回查**：看到版本表、更新时间、文件头部时先抽取原文；不要轻易写“未标注日期”。
5. **源与目标分开**：handover、`/tmp` 源文件、`~/.agents/skills/...`、项目仓文件是不同对象；同步前后要写路径和状态。
6. **Markdown 可渲染**：内嵌代码块时检查 fence 层级；需要嵌套三反引号时，外层用四反引号。
7. **能力不混淆**：Claude Code host 调用 Kimi 模型时，不默认拥有 Claude 原生 server tools；Kimi CLI/Web 的能力也不自动迁移到 Claude Code host。
8. **收尾给自检表**：多步交付最后列 `path / status / evidence`，例如 hash、diff 结论、命令摘要或源文档片段。

最小自检句式：

```text
自检：
- host/tool：...
- source artifact：...
- installed/runtime target：...
- 已验证：...
- 未验证/待裁决：...
```

## 外部审稿标准

优先查这些硬问题：

1. **证据链**：结论是否带源文件、原文片段、读取范围和规则映射。
2. **量化自证**：出现百分比、数量级、覆盖范围时，要有统计方法；否则改成定性表述。
3. **真相层一致**：报告正文、源文件、安装目标、运行态投影是否一致；必须给路径、hash 或 diff 结果。
4. **时间事实**：凡称“未标注日期”或给出年份范围，先抽取源文件头部验证。
5. **可安装性**：frontmatter、文件名、代码块 fence、标准字段、目标目录是否符合当前 skill / repo 规范。
6. **状态口径**：区分“已写入报告”“已生成源文件”“已安装”“已被 host 读取验证”。
7. **边界声明**：只读前几页、只抽样、只自评时必须明说，不得升级成全量结论。

## 外部调教方式

对 Kimi 的反馈应短、硬、可验证：

- 只给 2-5 个明确修正点，避免要求重写整篇。
- 每个修正点写清“错在哪里 / 应改成什么 / 验证方式”。
- 要求新增一节自检结果，列 `path / status / evidence`。
- 要求先修事实层和产物一致性，再谈风格润色。
- 如果它声明“已同步 / 已安装 / 已验证”，要求给出真实路径、hash、diff 或命令输出摘要。

## 典型纠偏模板

```text
只做纠偏，不要重写整篇。

必须改：
1. [事实错误] 对照源文件重新抽取，给文件名和原文证据。
2. [量化声明] 若无统计，改成定性；若保留数字，给抽样方法。
3. [真相层] 明确报告正文、源文件、安装目标、运行态投影四者状态。
4. [可渲染性] 检查 Markdown fence / frontmatter / 标准字段。

最后新增“自检结果”：列 path、hash 或 diff 结论，不要只写“已完成”。
```

## 质量分级

| 级别 | 标准 |
|---|---|
| A | 证据充分、边界清楚、源文件与安装目标一致、能自证修订 |
| B | 结构好、能吸收反馈，但需外部指出事实层或同步层问题 |
| C | 主要是风格总结，缺证据、缺验证、产物未闭环 |
| D | 把计划说成完成，或混淆 host / model / source / runtime |

## 已知倾向

从 2026-04-29 `sensoro-prd-writer` handover 审稿看，Kimi 的优点是结构化强、能吸收反馈、能把泛问题改成可裁决问题；弱点是真相层措辞容易不够精确，尤其是：

- 把“报告已改”说成“源文件已同步”。
- 把“目标路径存在但未覆盖”说成“尚未安装”。
- 对日期、读取范围、量化比例的回查不够细。
- Markdown 内嵌代码块可能未按渲染层验证。

后续调教重点：要求它每次区分 `报告 / 源文件 / 安装目标 / host 可见性`，并用路径与证据收口。
