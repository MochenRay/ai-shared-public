# Handover 写作规范

## 目标

`handover.md` 只负责一件事：

**让下一个模型在最短时间内知道“现在是什么状态，接下来该做什么”。**

它不是项目档案总表，不是阶段日志，不是冻结决策全集，也不是文件索引大全。

## 默认定位

把 `handover.md` 视为：

- 当前快照
- 当前边界
- 下一步入口

把下面这些内容移到别处：

- 稳定定位与长期边界：`project.md`
- 历史推进过程：`changelog/YYYY/YYYY-MM.md`
- 阶段任务拆解：`plans/*.md`
- 长期参考与冻结合同：`reference/*.md`
- 一次性会话判断：`observations/*.md`
- 运行手册：`runbooks/*.md`

## 推荐结构

`handover.md` 默认保持 6 个部分以内：

1. 标题与时间元信息
2. 一句话状态
3. 当前已验证事实
4. 当前边界与未完成项
5. 下一步
6. 推荐读取顺序 / 相关文件

如果某一块已经膨胀，就说明它应该被拆到别的文件。

## 每部分该写什么

### 1. 标题与时间元信息

保留：

- 更新日期
- 最近确认时间
- 最近编辑方
- 当前阶段（可选）

不要写：

- 长段背景叙述
- 多轮历史摘要

### 2. 一句话状态

只写当前判断，不写整个项目故事。

示例：

- `Phase 2 已完成，下一步拆 Phase 3 task plan。`
- `主链仍未打通，当前先修数据合同，再继续页面接骨。`

### 3. 当前已验证事实

只放“现在仍成立”的高信号事实，建议控制在 4-8 条。

优先写：

- 当前在哪个阶段
- 最近一次验证结论
- 当前可信基线是什么
- 哪个仓库 / 分支 / worktree 可以继续干活

不要把 `T00-T26` 全部复述一遍。

### 4. 当前边界与未完成项

只写现在真的会影响下一步判断的边界，建议控制在 3-6 条。

优先写：

- 当前还没做什么
- 哪些“看起来完成了”但其实不是
- 哪些环境限制会影响验证

### 5. 下一步

只写最近 1-3 个动作。

格式建议：

1. 先做什么
2. 再做什么
3. 不要回头做什么

### 6. 推荐读取顺序 / 相关文件

只保留最小必需跳转，建议控制在 4-8 个入口。

优先级建议：

1. `project.md`
2. 当前阶段的 `plan`
3. 最近的验证文档
4. 当前项目 `changelog`
5. 只有在需要时才读 `reference/*.md`

不要把整个项目目录树都抄进来。

## 明确禁止

`handover.md` 里默认不要出现：

- 完整任务历史流水账
- 冻结决策全文复写
- 大段验证命令明细
- 超长文件清单
- 原始对话回放
- “所有你可能需要知道的一切”

这些内容都应该外移。

## 行数纪律

- 目标长度：`60-120` 行
- 超过 `120` 行：开始重构
- 超过 `160` 行：默认视为失控，必须拆分

`handover` 的约束应该严于普通 Markdown，因为它是冷启动高频入口。

## 改写原则

更新 `handover.md` 时，优先做“改写当前快照”，而不是继续在旧快照后面叠加。

允许追加的内容只有：

- 新的确认日期
- 新的当前状态
- 新的下一步

历史变化一律进 `changelog`。

## 推荐模板

```md
# <项目名> 交接页
> 更新日期：YYYY-MM-DD
> 最近确认时间：YYYY-MM-DD（时区）
> 最近编辑方：<model>
> 当前阶段：<phase>

## 一句话状态
- <当前状态>

## 当前已验证事实
- <事实 1>
- <事实 2>
- <事实 3>

## 当前边界
- <边界 1>
- <边界 2>

## 下一步
1. <动作 1>
2. <动作 2>

## 推荐读取顺序
1. `project.md`
2. `<当前阶段计划>`
3. `<最近验证文档>`
4. `changelog/YYYY/YYYY-MM.md`
```
