# Task 分阶段切片

## Purpose

记录复杂任务如何拆成可恢复、可继续执行的阶段，而不是把临时任务目录或锁文件当成长久知识。

## Source Boundary

这个模式来自对 Claude 旧 `tasks/` 归档的提炼：多数目录只有 `.lock` 与 `.highwatermark`，只有少量 JSON 阶段任务具备长期价值。

可提升的是“阶段切片表达法”，不是任务目录机制本身。

## Stable Shape

- `subject`：阶段标题，应该一眼看出本阶段产出。
- `description`：阶段边界，说明本阶段交付什么，不展开完整实现方案。
- `activeForm`：当前动作表述，便于中途恢复和过程汇报。
- `status`：当前阶段状态，例如 `pending`、`in_progress`、`completed`。
- `blocks`：本阶段会阻塞哪些后续任务。
- `blockedBy`：本阶段依赖哪些前置条件。

## Slicing Principles

- 标题写产出，不写抽象口号。
- 描述写边界，不展开全部实现细节。
- 能写依赖时就显式写 `blocks` / `blockedBy`。
- 运行时状态和长期记忆分开保存，不混放。
- 计划文件应该帮助恢复执行，不应该膨胀成完整设计稿。

## Good Default Progression

1. 项目初始化或地基
2. 全局布局或共享框架
3. 关键入口页或主入口
4. 组件簇或子系统簇
5. 清理、补全或占位页

前端类任务的常见顺序是：环境与设计系统 -> 全局布局 -> 关键入口页 -> 通用展示框架 -> 具体组件簇 -> 补全占位页。

## Codex Execution Default

复杂任务需要可恢复切片时，默认写清三层：

- 标题：本阶段产出是什么。
- 范围：本阶段交付边界是什么。
- 动作：当前正在做什么，下一步如何恢复。

能表达依赖时，同时写 `blocks` / `blockedBy`。如果依赖关系还说不清，先在正文里明确“待确认”，不要伪造结构化依赖。

## Promotion Gate

- 提升的是切片方法，不是某一轮任务目录本身。
- `.lock`、`.highwatermark` 等运行时元数据继续留在本地运行层。
- 单次项目编号、历史流水和没有跨任务复用价值的具体任务，不进入 shared playbook。
- 只有能帮助下一次更快拆阶段、恢复依赖关系的方法，才值得进入 shared playbook。
