# 共享层规则

## 用途

当你需要判断以下问题时，读取本文件：

- 新的长期文档应该放在哪里
- 本地材料应该提升到共享层、做成本地投影副本，还是继续留在本地
- 本地内容和共享层内容不一致时该更新哪一侧
- 一次修改应该追加、改写，还是暂缓

## 默认读取规则

- 最小启动只读 `README.md` 和 `agents/<model>.md`。
- 第一次写回共享层前，或范围与边界不清时，再读本文件。
- 项目、画像、方法手册、延续线索和共享记忆按需读取。

## 上下文读取失败阻断

- 当冷启动、项目延续、写回前置或用户明确要求读取某个文件/目录，但因权限、路径、sandbox、include directory、文件缺失或工具限制未能读到具体内容时，必须先停止依赖该上下文的原任务，并向用户说明缺失路径、失败原因、已知/未知边界和可选修复方式。
- 不得把“知道有这条规则/路径”当作“已读取内容”；不得在关键上下文缺失时继续做业务代码、项目状态或共享规则判断。
- 配置修复、workspace/includeDirectories/trust/sandbox 白名单变更后，必须用工具验证实际读取结果，再恢复原任务；不能只根据配置已写入宣称已加载共享内容。

## 正式边界

### `AI-Shared` 负责

- 长期 Markdown 知识和文档
- 稳定事实与已确认决策
- 更详细的用户画像与探索材料
- 项目首页、操作手册、观察记录和交接页
- 可复用的方法手册与提示模式
- 共享规则与长期模型适配说明
- 本地入口投影所依赖的来源文件

### 本地目录负责

- 原始历史与运行态
- sqlite/jsonl 会话存储
- 工具缓存与插件运行态
- 当前生效设置、钩子、计划文件、锁文件
- 工具坚持自行拥有的自动生成文件

## 文档与运行态的区分

- 按制品类型划分，不按“摘要”还是“详细”划分。
- 只要是长期文档，即使很详细，也属于 `AI-Shared`。
- 运行态即使很有用，仍然留在工具本地目录。
- 如果工具为了启动必须保留本地规则或入口副本，把它视为投影副本，而不是正式来源。

## 默认归档位置

- 新的长期 Markdown 文档默认创建在 `AI-Shared`。
- 只有在内容属于运行态，或工具明确要求本地投影副本时，才新建本地文件。
- 如果因为工具限制先在本地创建了长期文档，应尽快回收进 `AI-Shared`。
- 如果还不确定该放哪，就把问题登记到 `memory/open-questions.md`，先暂缓，不要硬塞进错误目录。

## 并行执行默认策略

- 对中大型实现、跨模块改造、可并行的排障或联调任务，默认优先使用 `git worktree + agents 集群` 并行推进。
- 如果目标已经是 Git 仓库，默认路径是：主线程先冻结共享合同，再拆 `worktree` 与 `agent` 边界，随后并行执行。
- 如果目标还不是 Git 目录，默认路径是：主线程先冻结共享合同和目录边界，再直接使用 `agents` 集群并行；是否 Git 化，放到收口后统一决定。
- 共享合同至少要写清：目标、成功标准、真相层、边界、并行边界、集成顺序。
- 主线程负责冻结共享合同、拆分 `worktree` 与 `agent` 边界、集成结果、解决冲突、做最终验证。
- 子 `agent / worktree` 只在独立边界内实现或排查，不覆盖真相层和未合并改动，并返回改动范围、验证结果和未解决事项。
- 并行是默认行为，不需要等用户再次提醒；如果不启用集群，主线程应明确写出例外原因。
- 主线程必须先定义边界，再派发子任务；子线程不能自发扩边界。
- 最终只有主线程有权宣布本轮收口完成。

## 阶段状态口径

- 冻结、执行、验证和后续增量是不同状态，不得合并表述。
- 计划或合同已冻结，只能说“已冻结、待执行”，不能说“已完成”或暗示已经落地。
- 只有有代码/文档改动、运行结果或验收证据时，才能说“已执行”；只有验证证据通过时，才能说“已完成”。
- 当出现 `v2`、`Phase 8`、补丁轮或用户新增问题时，先说明它与既有 `v1/当前 phase` 的关系：是追加需求、替代范围、合并执行，还是后续阶段。
- 回答“做完了吗”“继续哪个 phase”“v2 是否包含 v1”这类问题前，先对照最新 `plan / verification / handover / git HEAD`，不要把计划存在误当成执行完成。
- 典型反例：`v1` 已冻结但未执行，用户又补了 `v2` 输入材料；不能默认跳去 `v2`，应先说明 `v1` 仍是默认下一步，除非用户明确改序。
- 如果发现自己把阶段状态说混，立即纠正并把后续执行顺序重新分清，不沿着错误口径继续推进。

## 默认表达压缩协议

- Codex、Claude、Gemini、Kimi 默认启用 `caveman` 表达压缩，默认模式为 `wenyan-full`。
- 目标是减少废话和输出 token，保留事实、证据、边界、风险、命令、路径、代码和验证结果。
- 这是表达层规则，不得用来省略必要推理交代、风险确认、测试证据、文件引用或用户要求的结构。
- 生成或修改文档前，先判断受众：给用户阅读的计划、路线图、复盘、报告、PRD、交接和说明文档，正文必须使用中文。
- 给 AI、工具或机器消费的文件可按生态使用英文或中英混合；但其中面向用户阅读的段落、摘要和结论仍默认中文。
- 混合受众文档采用中文为主；技术名词、路径、命令、配置键、错误原文、代码、schema 字段和引用标题保持原样。
- 若受众不清，默认按用户阅读处理，先写中文；只有明确是工具协议、代码接口或模型内部提示时才允许英文为主。
- 用户说“关闭 caveman”“normal mode”“普通模式”“不用文言”或同义表达时，本轮会话按用户要求切换；用户可随时重新开启。
- 涉及高风险操作、法律/医疗/财务等高 stakes 说明、不可逆确认、复杂步骤容易误读、或用户要求解释清楚时，先用清晰现代中文说明；该段结束后恢复默认模式。
- 代码、命令、文件路径、报错原文、引用文本、commit message 和 PR 正文按任务需要保持常规准确格式，除非用户明确要求压缩。

## 默认不启用集群的例外

- 任务很小，单线程更快。
- 改动高度耦合，拆开只会增加冲突。
- 当前阻塞点是单一事实核查或单机实验，无法并行。

## 本地变更如何同步到共享层

- 本地长期文档发生变化时，先判断是否需要登记 `handoff/sync-queue.md`。
- 适合登记的情况包括：本地规则、本地项目记忆、本地长期手册、本地入口投影副本发生了可能影响共享正式来源的变化。
- 不适合登记的情况包括：历史、sqlite/jsonl、缓存、计划文件、临时调试、一次性任务记录、纯运行态修补。
- `sync-queue.md` 只记录短条目，不放正文；正文应继续留在正式文档或本地原文件中。
- 如果共享目标还不清楚，可以先把“同步目标”记为待判断；如果事实本身未核实，先不要同步。
- 完成判断后，把条目移动到“已同步”或“已暂缓”，不要长期挂在待判断区。

## 语义同步协议

- 按语义同步，不做盲目的全文镜像。
- 先比较本地源材料与该主题当前共享层正式文档的差异。
- 当本地长期笔记出现稳定增量时，更新共享层正式文档。
- 本地运行噪音不要写进共享层。
- 如果因为启动原因存在本地投影副本，在共享层版本确认后再刷新本地副本。
- 本地和共享层不一致时，更新应该承担正式来源职责的那一层。
- 如果差异仍然说不清，就记到 `handoff/sync-queue.md` 或 `memory/open-questions.md`，先暂缓。

## 写后检查

- 每次写回共享层后，都要查看统一守卫脚本 `scripts/validate-shared-writeback.sh` 的结果。
- 如果出现路径错误、疑似 transcript/runtime dump、或行数提醒，当轮处理，不留到下轮。
- 共享层的写后检查优先用于纠正文档落点和文档形态，不用于替代内容判断。

## 投影副本规则

- 正文规则通常不应在多个地方重复维护。
- 只有当工具为了自动加载或启动行为明确要求时，才保留本地投影副本。
- 投影副本应尽量短小、标注清楚，并且不手动当成正式来源维护。
- 优先采用“共享层一份正式规则 + 适配文件里的少量模型差异”，不要维护三套彼此漂移的独立规则系统。
- 如果用户新增的是通用规章、长期协作要求或跨模型都应遵守的流程，不要只更新当前模型适配页；必须同步检查并更新 `agents/codex.md`、`agents/claude.md`、`agents/gemini.md`、`agents/kimi.md` 中相关入口，除非该规则明确只适用于单一模型。
- Agent Skills 安装/更新默认按跨模型 universal skill 处理，真相层为 `~/.agents/skills/`；仅用户明确指定单端或工具原生例外时才写单一工具 home。细则见 `rules/skills-governance.md`。

## 写回位置

- 共享规则 -> `rules/`
- 用户事实、画像、健康和探索材料 -> `profile/`
- 跨项目记忆与已确认决策 -> `memory/`
- 项目定位、操作手册、观察记录和当前状态 -> `projects/<name>/`
- 可复用的方法与稳定流程模式 -> `playbooks/`
- 值得继续的未完成线索 -> `continuations/`
- 本地变更待判断事项 -> `handoff/sync-queue.md`
- 短期协作事项 -> `handoff/inbox.md`
- 全局共享层变更历史 -> `handoff/changelog/YYYY/YYYY-MM.md`
- 项目内推进历史 -> `projects/<name>/changelog/YYYY/YYYY-MM.md`
- 根入口和本地入口来源文件 -> `entrypoints/`

## 项目 handover 规则

- `projects/<name>/handover.md` 只保留当前快照、当前边界和下一步入口。
- 稳定定位写进 `project.md`，历史推进写进 `changelog/`，任务拆解写进 `plans/`，冻结合同写进 `reference/`。
- `handover.md` 默认控制在 `60-120` 行；超过 `120` 行开始重构，超过 `160` 行默认必须拆分。
- 更新 `handover.md` 时优先改写当前快照，不要继续往里面堆完整历史。
- 具体结构与模板统一参考 `playbooks/handover-writing.md`。

## 行数提醒

- Markdown 文档超过 200 行时，要判断是拆分、压缩，还是暂时保留。
- 超过 300 行时，默认优先拆成“母文件 + 子文件”。
- 优先做渐进式加载，不默认删减内容。
- 频繁读取、承担索引职责、或自动加载频率高的文件，更应尽早拆分。

## 新增结构规则

- 优先在已有根目录下增加子目录，不要轻易新建顶层根目录。
- 新增根目录或新的子目录约定时，要同时更新 `manifest.md`。
- 准备批量使用新的路径形态前，先更新守卫脚本识别的白名单。

## 日期规则

- 事实、决策、状态如果对时间敏感，就写明确日期。
- 使用当前工作时区下的自然日日期。
- 容易过期的快照应在顶部标注“最近确认时间”之类的中文提示。

## 不确定性标记

- 对不确定内容明确标注“未核实”“暂定”“待确认”“最近确认时间”等字样。
- 不要把不确定材料悄悄升级成稳定记忆。

## 冲突处理

- 在完成核实前，保留上一版稳定记录。
- 有争议的内容先记到 `memory/open-questions.md` 或 `handoff/sync-queue.md`。
- 同时把变更写入对应的月志；如果新增了新的月份文件或新的项目日志入口，再更新 `handoff/changelog.md`。
- 只有在完成核实或用户确认后，才更新正式事实。

## 追加还是改写

- 新事件、新决策、新日期快照，采用追加。
- 收紧正式摘要、重组长期文档、纠正已核实错误，采用改写。
- 不要借着改写之名偷偷塞入新的解释。
- 除非细节已经在 `AI-Shared` 的别处保留，或被明确保存在本地投影副本中，否则不要只用摘要替换细节。

## 防熵规则

- 不放原始聊天记录。
- 不放隐藏推理倾倒。
- 不放 sqlite/jsonl/缓存/运行态制品。
- 不把巨型滚动日志堆进单文件。
- 不把同一份规则正文在多个正式来源里失控复制。
